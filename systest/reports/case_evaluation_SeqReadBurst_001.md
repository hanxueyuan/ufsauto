# 测试用例评估报告：t_perf_SeqReadBurst_001

**评估日期**：2026-03-23  
**评估对象**：`suites/performance/t_perf_SeqReadBurst_001.py`（顺序读 Burst 性能测试）  
**评估维度**：场景覆盖度 + 断言充分性

---

## 1. 原始 Case 分析

### 1.1 测试意图

验证 UFS 设备顺序读 Burst 性能是否达标（≥2100 MB/s）。使用 FIO 工具，128K block size，sync 引擎，运行 60 秒。

### 1.2 场景覆盖矩阵

| 场景维度 | 应覆盖 | 当前状态 | 说明 |
|---------|--------|---------|------|
| **Block Size** | 4K / 128K / 256K / 1M | ❌ 仅 128K | 不同 bs 性能特征差异巨大，128K 只是一个点 |
| **IO 引擎** | sync / libaio / io_uring | ❌ 写死 sync | sync 引擎无法测试异步 IO 路径，且不是生产环境典型配置 |
| **Queue Depth** | QD1 / QD4 / QD16 / QD32 | ❌ 未指定（默认 QD1） | 顺序读在不同队列深度下表现不同 |
| **Direct IO** | direct=1 / buffered | ⚠️ 未显式指定 | FIOConfig 默认 direct=True，但 case 没显式控制 |
| **Ramp Time** | 有 ramp_time | ❌ 无 | 没有预热期，前几秒的冷启动数据会拉低平均值 |
| **数据预填充** | 先写后读 vs 新文件 | ❌ 无预填充 | 读空文件（或 sparse file）和读真实数据性能不同 |
| **Time-based** | runtime 固定时长 | ⚠️ 依赖 FIO wrapper | 未在 case 层面显式确认 |
| **数据校验** | verify=md5/crc32 | ❌ 无 | 没有数据完整性校验，只看速度不看正确性 |
| **多次运行取平均** | ≥3 次取中位数 | ❌ 单次运行 | 单次结果波动大，不可靠 |
| **稳态测试** | 长时间运行看衰减 | ❌ 无 | 60 秒可能还在 SLC cache 阶段 |
| **并发任务** | numjobs > 1 | ❌ 无 | 多线程顺序读的表现未测 |
| **文件大小** | 多种 size | ❌ 仅 1G | 小文件可能全在 cache 里 |

**覆盖率**：12 个维度中，0 个完整覆盖，2 个部分覆盖 → **约 8%**

### 1.3 断言清单

| 断言内容 | 状态 | 说明 |
|---------|------|------|
| 带宽 ≥ 2100 MB/s | ✅ 有 | 唯一的断言 |
| IOPS 在合理范围 | ❌ 缺失 | 有采集，没验证 |
| 平均延迟 < 上限 | ❌ 缺失 | 有采集，没验证 |
| p99.999 尾延迟 < 上限 | ❌ 缺失 | 有采集，没验证 |
| 数据完整性 | ❌ 缺失 | 完全没有 |
| CPU 使用率合理 | ❌ 缺失 | FIOMetrics 有 cpu 字段但未用 |
| 测试稳定性（多次一致） | ❌ 缺失 | 单次运行 |

**断言充分性**：7 项中仅 1 项 → **14%**

### 1.4 代码质量问题

1. **setup() 中 `import os` 放在方法内部**：不规范，应在文件头导入
2. **setup() 检查 `which fio` 但 execute() 通过 wrapper 调用**：检查和执行路径不一致
3. **硬编码 target=2100**：不同 UFS 规格（3.0/3.1/4.0）目标值不同，应可配置
4. **simulate 模式逻辑分散**：init/execute 里都有 if simulate 分支，耦合度高
5. **没有记录 FIO 的完整配置**：出问题时无法复现
6. **teardown 吞了异常**：`return True` 即使清理失败

### 1.5 原始评分

| 维度 | 分数 (1-10) | 说明 |
|------|------------|------|
| 场景覆盖度 | 2 | 只测了一个固定配置点 |
| 断言充分性 | 2 | 只验证了带宽，其他指标白采集了 |
| 代码质量 | 5 | 结构清晰但有硬编码和不规范 |
| 可维护性 | 4 | 参数不可配，改参数要改代码 |
| 可诊断性 | 5 | 有日志但缺 FIO 完整参数记录 |
| **综合** | **3.6** | 能跑但测试价值有限 |

## 2. 改进内容摘要

### 改进项一览

| # | 改进 | 原因 |
|---|------|------|
| 1 | 增加 `ramp_time=10` | 避免冷启动偏差 |
| 2 | 增加延迟、IOPS 等多维指标标注 | 采集了就应该标注是否达标 |
| 3 | **validate 改为"标注"模式，永远返回 True** | 性能测试不应因指标不达标而 FAIL——指标达不达标需要后续深入分析，不是测试本身的问题。FAIL 只用于测试跑不起来（ERROR）的情况 |
| 4 | 增加 `result['annotations']` 字段 | 所有指标的达标情况结构化记录，供报告层/分析层消费 |
| 5 | 增加 FIO 完整配置日志 | 出问题可复现 |
| 6 | target/延迟上限等通过 `__init__` 参数可配 | 支持不同规格设备 |
| 7 | setup 中预填充测试文件 | 避免读 sparse file |
| 8 | 增加 `verify` 选项（可选） | 需要时可开启数据校验 |
| 9 | 修复 `import os` 位置 | 代码规范 |

### 未改但建议后续做的

- **多次运行取中位数**：需要改 runner 框架支持，单个 case 不好做
- **多 block size 参数化**：建议新建 `t_perf_SeqReadBurst_002.py`（4K）等，而不是塞在一个 case 里
- **libaio/io_uring 引擎测试**：同上，建议独立 case
- **稳态测试（10 分钟+）**：建议独立 case，放在 `suites/endurance/`

## 4. 进一步建议

### 对这个 Case

1. **增加多次运行模式**：在 runner 框架中增加 `repeat` 支持，自动取中位数
2. **增加 baseline 对比**：记录历史结果，检测性能回退
3. **参数化测试**：用 pytest 的 `@pytest.mark.parametrize` 或自定义装饰器，一个文件测多种配置

### 对整个测试框架

1. **所有 7 个 case 都有同样的问题**：setup 大量重复、断言单薄、参数硬编码
2. **缺少功能测试 suite**：当前只有 performance 和 qos，缺 `functional/`（读写正确性、Trim、错误处理）
3. **缺少异常/边界测试**：满盘、零长度 IO、非法参数、中断恢复
4. **缺少回归测试**：没有 known-bug 对应的 case
5. **FIO 配置应外置**：用 YAML/JSON 定义测试矩阵，case 只负责执行和验证逻辑
6. **建议增加 test tag/label 机制**：便于按类别（perf/qos/func/stress）选择性执行
