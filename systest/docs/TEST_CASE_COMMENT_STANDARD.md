## 💡 Precondition 核心要点

### 为什么 Precondition 如此重要？

| 要点 | 目的 | 价值 |
|------|------|------|
| **系统环境收集** | 记录测试时的系统状态 | 确保测试可复现，问题可追溯 |
| **测试目标信息** | 明确测试对象 | 避免测试错误设备 |
| **存储设备配置** | 记录功能开关状态 | 确保测试条件一致 |
| **LUN 配置检查** | 了解器件结构 | 避免 LUN 配置影响测试结果 |
| **健康状况检查** | 确保器件正常 | 避免带病测试，结果不可信 |
| **前置条件验证** | 强制检查必要条件 | 不满足条件则跳过，避免无效测试 |

### Postcondition 的配置恢复原则

**原则**：尽量恢复测试前的状态

| 场景 | 操作 | 说明 |
|------|------|------|
| 修改了设备配置 | 恢复原配置 | 如修改了 TURBO Mode，测试后恢复 |
| 关闭了功能 | 重新开启 | 如关闭了自动 GC，测试后开启 |
| 创建了测试文件 | 清理文件 | 释放磁盘空间 |
| 修改了系统参数 | 恢复原参数 | 如修改了 IO 调度器 |
| 设备处于忙碌状态 | 等待空闲 | 确保不影响后续测试 |

## 📝 完整示例（更新版）

### 示例 1：性能测试用例（包含完整 Precondition）

```python
"""
测试名称：t_performance_SequentialReadBurst_001

测试目的：
验证 UFS 设备的顺序读带宽 Burst 性能，评估设备在短时间内能达到的
最大读取带宽，确保满足车规级 UFS 3.1 的≥2100 MB/s 要求。

测试流程：
1. Precondition（前置条件）
   1.1 系统环境收集
       - 操作系统：Debian 12, kernel 5.15.120
       - CPU/内存：8 核，16GB
       - FIO 版本：fio-3.33
   
   1.2 测试目标信息收集
       - 设备路径：/dev/ufs0
       - 设备型号：UFS 3.1 128GB
       - 固件版本：v1.0.0
       - 设备容量：128GB
       - 可用空间：≥10GB
   
   1.3 存储设备配置检查
       - 开启功能：TURBO Mode（提升峰值性能）
       - 关闭功能：省电模式（避免性能限制）
       - 特殊配置：无
   
   1.4 UFS 器件配置检查
       - LUN 数量：4 个
       - LUN0：64GB 系统盘（已挂载）
       - LUN1：32GB 数据盘（测试目标）
       - LUN2：16GB 日志盘
       - LUN3：16GB 预留
       - LUN 映射：LUN1→/dev/ufs0
   
   1.5 器件健康状况检查
       - SMART 状态：正常
       - 剩余寿命：98%
       - 坏块数量：0
       - 温度状态：35℃（当前）/ 45℃（最高）
       - 错误计数：CRC 错误=0, 重传次数=0
   
   1.6 前置条件验证
       - ✓ SMART 状态必须为正常
       - ✓ 可用空间必须≥10GB
       - ✓ 温度必须<70℃
       - ✓ 剩余寿命必须>90%

2. Test Steps（测试步骤）
   - 步骤 1: 使用 FIO 工具发起顺序读测试
   - 步骤 2: 配置参数：
     * rw=read（顺序读模式）
     * bs=128k（块大小 128KB，模拟大文件读取）
     * iodepth=32（队列深度 32，充分利用 NCQ）
     * numjobs=1（单线程，避免 CPU 瓶颈）
     * runtime=60（运行 60 秒，Burst 测试）
     * time_based（基于时间的测试，确保测试时长准确）
   - 步骤 3: FIO 持续读取 60 秒，记录带宽数据
   - 步骤 4: 收集测试结果，计算平均带宽

3. Postcondition（后置条件）
   - 测试结果保存到 results/performance/目录
   - 配置恢复：无配置变更，无需恢复
   - 设备恢复到空闲状态（等待 5 秒）
   - 数据清理：无测试数据残留

测试参数：
- rw: read（顺序读）
- bs: 128k（块大小）
- iodepth: 32（队列深度）
- numjobs: 1（线程数）
- runtime: 60（测试时长，秒）
- time_based: true（基于时间）

验收标准：
- PASS: 平均带宽 ≥ 2100 MB/s（允许 5% 误差，即≥1995 MB/s）
- FAIL: 平均带宽 < 1995 MB/s

注意事项：
- Burst 测试时间短（60 秒），反映设备峰值性能
- 测试前确保设备未处于过热状态
- 如果测试失败，检查设备温度、队列深度配置
- 建议重复测试 3 次取平均值
"""
```

### 示例 2：可靠性测试用例（包含配置恢复）

```python
"""
测试名称：t_reliability_StabilityTest_001

测试目的：
验证 UFS 设备在长时间（24 小时）连续工作下的稳定性，
确保无数据错误、无性能衰减、无设备故障，满足车规级可靠性要求。

测试流程：
1. Precondition（前置条件）
   1.1 系统环境收集
       - 操作系统：Debian 12, kernel 5.15.120
       - CPU/内存：8 核，16GB
       - FIO 版本：fio-3.33
   
   1.2 测试目标信息收集
       - 设备路径：/dev/ufs0
       - 设备型号：UFS 3.1 128GB
       - 固件版本：v1.0.0
       - 设备容量：128GB
       - 可用空间：≥20GB（长时间测试需要更多空间）
   
   1.3 存储设备配置检查
       - 开启功能：无特殊开启
       - 关闭功能：自动休眠（避免影响长时间测试）
       - 特殊配置：IO 调度器设置为 none（减少调度延迟）
   
   1.4 UFS 器件配置检查
       - LUN 数量：4 个
       - LUN0：64GB 系统盘
       - LUN1：32GB 数据盘（测试目标）
       - LUN2：16GB 日志盘
       - LUN3：16GB 预留
       - LUN 映射：LUN1→/dev/ufs0
   
   1.5 器件健康状况检查
       - SMART 状态：正常
       - 剩余寿命：98%
       - 坏块数量：0
       - 温度状态：35℃（当前）/ 45℃（最高）
       - 错误计数：CRC 错误=0, 重传次数=0
   
   1.6 前置条件验证
       - ✓ SMART 状态必须为正常
       - ✓ 可用空间必须≥20GB
       - ✓ 温度必须<70℃
       - ✓ 电源必须稳定（建议使用 UPS）
       - ✓ 散热条件良好（建议加散热片或风扇）

2. Test Steps（测试步骤）
   - 步骤 1: 启动 24 小时稳定性测试
   - 步骤 2: 配置参数：
     * rw=randrw（随机混合读写，70% 读 30% 写）
     * bs=4k（4KB 块大小，模拟真实负载）
     * iodepth=32（队列深度 32）
     * numjobs=2（2 个并发线程，模拟多任务）
     * runtime=86400（运行 24 小时）
     * verify=0（不校验数据，减少 CPU 负载）
   - 步骤 3: 每 5 分钟记录一次性能数据：
     * 带宽（MB/s）
     * IOPS
     * 延迟（μs）
     * 设备温度（℃）
   - 步骤 4: 监控错误计数：
     * CRC 错误
     * 重传次数
     * IO 错误
   - 步骤 5: 24 小时后停止测试，分析数据

3. Postcondition（后置条件）
   - 测试结果保存到 results/reliability/目录
   - 配置恢复：
     * 恢复自动休眠功能
     * 恢复 IO 调度器为默认值（如 mq-deadline）
   - 设备恢复到空闲状态（等待 10 秒）
   - 数据清理：删除测试生成的临时文件

测试参数：
- rw: randrw（混合读写）
- rwmixread: 70（70% 读，30% 写）
- bs: 4k（块大小）
- iodepth: 32（队列深度）
- numjobs: 2（并发线程）
- runtime: 86400（24 小时，秒）

验收标准：
- 无 IO 错误（错误计数=0）
- 性能衰减 < 20%（初始带宽 vs 最终带宽）
- 设备温度 < 70℃（全程）
- 无设备掉线或重启

注意事项：
- 24 小时测试时间长，确保电源稳定
- 测试前备份重要数据（虽然不会破坏数据）
- 建议定期检查设备温度
- 如温度超过 70℃，暂停测试并改善散热
- 测试过程中不要中断，否则需要重新开始
"""
```

## ✅ 注释检查清单

在提交测试用例前，检查注释是否包含：

### Precondition 检查（6 项）
- [ ] **系统环境收集** - 操作系统、CPU/内存、FIO 版本
- [ ] **测试目标信息** - 设备路径、型号、固件、容量
- [ ] **存储设备配置** - 功能开关状态、特殊配置
- [ ] **LUN 配置检查** - LUN 数量、各 LUN 配置、映射关系
- [ ] **健康状况检查** - SMART、寿命、坏块、温度、错误计数
- [ ] **前置条件验证** - 必须满足的条件列表

### Test Steps 检查（3 项）
- [ ] **步骤详细** - 每一步都有详细描述
- [ ] **参数明确** - 所有参数及说明
- [ ] **预期结果** - 关键步骤有预期结果

### Postcondition 检查（4 项）
- [ ] **结果保存** - 测试结果保存位置
- [ ] **配置恢复** - 如有配置变更，已恢复原状
- [ ] **设备恢复** - 设备恢复到空闲状态
- [ ] **数据清理** - 清理测试数据（如需要）

### 其他检查（3 项）
- [ ] **测试参数** - 所有参数列表及说明
- [ ] **验收标准** - 明确的 PASS/FAIL 标准
- [ ] **注意事项** - 重要提示和风险提示
