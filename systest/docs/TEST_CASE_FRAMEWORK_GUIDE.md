## 🎯 测试案例的核心要素

一个完整的系统测试案例必须包含以下**7 个核心部分**：

```
1. 测试目的（Why）
2. Precondition（Before）
3. Test Steps（How）
4. Postcondition（After）
5. 测试参数（What）
6. 验收标准（Pass/Fail）
7. 注意事项（Tips）
```

---

## 📋 各部分详细说明

### 1. 测试目的（Why）

**回答的问题**：
- 这个测试验证什么功能？
- 确保满足什么车规级要求？
- 为什么要做这个测试？

**写作要点**：
- ✅ 用一句话清晰描述
- ✅ 包含验证对象和指标
- ✅ 说明车规级要求

**示例**：
```python
测试目的:
验证 UFS 设备的顺序读带宽 Burst 性能，评估设备在短时间内能达到的最大读取带宽，
确保满足车规级 UFS 3.1 的≥2100 MB/s 要求。
```

---

### 2. Precondition（Before）

**回答的问题**：
- 测试前需要收集哪些信息？
- 如何收集这些信息（命令/方法）？
- 期望得到什么结果？
- 哪些条件必须满足？

**写作要点**：
- ✅ 说明**如何收集**（命令/文件/函数）
- ✅ 说明**预期结果**（阈值/范围）
- ✅ 说明**验证条件**（必须满足的条件）
- ❌ 不写死具体值

**结构**：
```python
Precondition:
1.1 系统环境收集
    - <信息类别>：<方法>，收集<字段>
    
1.2 测试目标信息收集
    - <信息项>：
      - 方法：<具体命令或文件路径>
      - 预期：<期望结果>
    
1.3 存储设备配置检查
    - 查看支持的功能：
      - 方法：<命令>
      - 预期：<结果>
    - 需要开启的功能：
      - <功能名>：
        - 检查方法：<命令>
        - 预期值：<值>
        - 开启方法：<命令>
    
1.4 UFS 器件配置检查
    - <配置项>：
      - 方法：<命令或函数>
      - 预期：<结果>
    
1.5 器件健康状况检查
    - <健康指标>：
      - 方法：<smartctl 命令或 sysfs 文件>
      - 预期：<阈值>
    
1.6 前置条件验证
    - ✓ <条件 1>（否则跳过测试）
    - ✓ <条件 2>（否则跳过测试）
```

**核心原则**：
- Precondition 是**测试方法说明**，不是**期望值列表**
- 让其他工程师能**按图索骥**执行相同的检查
- 确保测试**可复现**、**可追溯**

---

### 3. Test Steps（How）

**回答的问题**：
- 测试的具体步骤是什么？
- 每一步用什么命令和参数？
- 预期每一步得到什么结果？

**写作要点**：
- ✅ 步骤编号清晰（1, 2, 3...）
- ✅ 包含具体命令和参数
- ✅ 说明预期结果（如适用）

**示例**：
```python
Test Steps:
1. 使用 FIO 工具发起顺序读测试
   - 命令：fio --name=seq_read --filename=/dev/ufs0 --rw=read ...
2. 配置参数：rw=read, bs=128k, iodepth=32, numjobs=1, runtime=60, time_based
3. FIO 持续读取 60 秒，记录带宽数据
4. 收集测试结果，计算平均带宽
```

---

### 4. Postcondition（After）

**回答的问题**：
- 测试结果保存在哪里？
- 如何恢复测试前的配置？
- 如何清理测试数据？
- 如何检查测试后器件状态？

**写作要点**：
- ✅ 说明结果保存路径
- ✅ 说明配置恢复方法
- ✅ 说明数据清理方法
- ✅ 说明测试后状态检查

**示例**：
```python
Postcondition:
- 测试结果保存：results/performance/目录
- 配置恢复：
  - 恢复 TURBO Mode 为原始状态
  - 恢复省电模式为原始状态
- 设备恢复：等待 5 秒，让设备回到空闲状态
- 数据清理：
  - 删除测试生成的临时文件
  - 执行 fstrim /dev/ufs0 清理未使用块
- 测试后状态检查：
  - 重新读取 SMART 状态
  - 重新读取温度
  - 重新读取错误计数
  - 对比测试前后变化
```

---

### 5. 测试参数（What）

**回答的问题**：
- 测试使用了哪些关键参数？
- 每个参数的值是什么？
- 参数的含义是什么？

**写作要点**：
- ✅ 列出所有关键参数
- ✅ 说明参数值
- ✅ 简要说明参数含义

**示例**：
```python
测试参数:
- rw: read（读写模式：顺序读）
- bs: 128k（块大小：128KB）
- iodepth: 32（队列深度：32）
- numjobs: 1（并发线程数：1）
- runtime: 60（测试时长：60 秒）
```

---

### 6. 验收标准（Pass/Fail）

**回答的问题**：
- 如何判断测试通过？
- 具体的数值目标是什么？
- 允许的误差范围是多少？

**写作要点**：
- ✅ 明确 PASS 条件
- ✅ 明确 FAIL 条件
- ✅ 包含误差范围（如 5%）

**示例**：
```python
验收标准:
- PASS: 平均带宽 ≥ 2100 MB/s（允许 5% 误差，即≥1995 MB/s）
- FAIL: 平均带宽 < 1995 MB/s
```

---

### 7. 注意事项（Tips）

**回答的问题**：
- 有什么需要特别注意的？
- 有什么已知的限制或风险？
- 有什么优化建议？

**写作要点**：
- ✅ 至少 3 条实用建议
- ✅ 包含重复测试建议
- ✅ 包含故障排查建议

**示例**：
```python
注意事项:
- Burst 测试时间短（60 秒），反映设备峰值性能
- 测试前确保设备未处于过热状态
- 如果测试失败，检查设备温度、队列深度配置
- 建议重复测试 3 次取平均值
```

## 🔑 核心原则

### 可读性
- ✅ 让其他工程师能看懂
- ✅ 术语准确、表达清晰
- ✅ 结构清晰、层次分明

### 可复现性
- ✅ 命令完整、参数明确
- ✅ 步骤详细、可操作
- ✅ 预期结果清晰

### 可维护性
- ✅ 注释与代码一致
- ✅ 及时更新、保持同步
- ✅ 版本管理、变更可追溯
