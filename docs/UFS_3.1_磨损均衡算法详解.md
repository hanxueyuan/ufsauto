## 1. 为什么需要磨损均衡

### 1.1 NAND P/E 循环限制

NAND Flash 的每个块（Block）有有限的编程/擦除（P/E）循环次数：

| NAND 类型 | 每 Cell 位数 | 典型 P/E 循环 | 车规级 P/E 循环 |
|-----------|-------------|--------------|----------------|
| SLC | 1 bit | 50,000-100,000 | 100,000+ |
| MLC | 2 bits | 3,000-10,000 | 10,000+ |
| TLC | 3 bits | 1,000-3,000 | 3,000 |
| QLC | 4 bits | 500-1,000 | 不推荐车规 |

当某个块的 P/E 次数耗尽，该块将变成坏块，数据不再可靠。如果没有磨损均衡，频繁写入的"热"区域会远比"冷"区域更快耗尽寿命，导致设备过早失效。

### 1.2 不均匀磨损的后果

```
无磨损均衡时的擦除分布:

Block 0  (系统分区): ████████████████████ 50,000 次 → 已失效！
Block 1  (系统分区): ██████████████████── 45,000 次 → 即将失效
Block 2  (用户数据): ████────────────── 10,000 次
Block 3  (用户数据): ███─────────────── 8,000 次
Block 4  (冷数据):   █──────────────── 2,000 次
Block 5  (冷数据):   ───────────────── 500 次
...
设备寿命由最先失效的块决定！

有磨损均衡时:
Block 0: ████████████──── 30,000 次
Block 1: ████████████──── 29,500 次
Block 2: ████████████──── 30,200 次
Block 3: ███████████───── 28,000 次
Block 4: ████████████──── 29,800 次
Block 5: ████████████──── 30,100 次
→ 所有块均匀使用，设备寿命最大化
```

## 2. 动态磨损均衡（Dynamic Wear Leveling）

### 2.1 原理

动态 WL 在**写入新数据**时选择擦除次数最少的空闲块，确保热数据写入分散到不同块。

### 2.2 算法流程

```
写入请求到达
     │
     ↓
┌─────────────────────────┐
│ 1. 从 Free Block Pool   │
│    选择擦除次数最少的块   │
│                          │
│    排序: EC(B0)=100      │ ← 选择这个
│          EC(B1)=150      │
│          EC(B2)=200      │
│          EC(B3)=250      │
└────────┬────────────────┘
         ↓
┌─────────────────────────┐
│ 2. 将数据写入选中的块    │
│    Block B0, Page 0-N    │
└────────┬────────────────┘
         ↓
┌─────────────────────────┐
│ 3. 更新映射表            │
│    LPN → 新 PPN          │
└────────┬────────────────┘
         ↓
┌─────────────────────────┐
│ 4. 旧块中的数据标记无效  │
│    旧块等待 GC 回收      │
└─────────────────────────┘
```

### 2.3 数据结构

```c
/* 擦除计数表 */
typedef struct {
    uint32_t block_id;
    uint32_t erase_count;    /* 该块的累计擦除次数 */
    uint32_t last_erase_time; /* 最后一次擦除时间 */
} erase_count_entry_t;

/* Free Block Pool（按擦除次数排序的最小堆） */
typedef struct {
    erase_count_entry_t *heap;
    uint32_t size;
    uint32_t capacity;
} free_block_pool_t;

/* 获取擦除次数最少的空闲块 */
uint32_t get_min_ec_free_block(free_block_pool_t *pool) {
    return heap_extract_min(pool); /* O(log n) */
}
```

### 2.4 局限性

动态 WL 只能均衡**被写入的块**。如果某些块存储了很少更新的"冷数据"（如系统镜像），这些块的擦除次数会一直很低，而热数据块的擦除次数持续增长。这就需要静态 WL。

## 3. 静态磨损均衡（Static Wear Leveling）

### 3.1 原理

静态 WL 主动将**冷数据**从低擦除次数块迁移到高擦除次数块，释放低擦除块给热数据使用。

### 3.2 算法流程

```
定期检查（或达到阈值时触发）
     │
     ↓
┌─────────────────────────────────┐
│ 1. 扫描所有块的擦除计数         │
│    找到: 最大 EC 块 (Block_hot)  │
│          最小 EC 块 (Block_cold) │
│    差值: ΔEC = EC_max - EC_min   │
└────────────┬────────────────────┘
             ↓
┌─────────────────────────────────┐
│ 2. ΔEC > 阈值 (如 100)?        │
│    Yes → 执行数据交换            │
│    No  → 不操作                  │
└──┬──────────────────────┬───────┘
   │ Yes                  │ No
   ↓                      ↓ (结束)
┌─────────────────────────────────┐
│ 3. 数据交换流程                  │
│    a. 读取 Block_cold 的有效数据 │
│    b. 写入到空闲块 Block_free    │
│    c. 擦除 Block_cold            │
│    d. 将 Block_cold 加入热数据池 │
│    e. 更新映射表                 │
└─────────────────────────────────┘
```

### 3.3 触发条件与阈值

| 参数 | 典型值 | 说明 |
|------|--------|------|
| **EC 差值阈值** | 100-500 | ΔEC > 阈值时触发交换 |
| **检查周期** | 每 1000-10000 次写入 | 定期扫描 |
| **最大交换频率** | 每分钟 1-5 次 | 限制对性能的影响 |
| **冷数据判定** | 超过 N 次 GC 周期未被更新 | 识别冷数据块 |

### 3.4 对性能的影响

静态 WL 需要额外的数据搬移操作：

```
一次静态 WL 交换的开销:
  1. 读取冷数据块: 256 pages × 50μs = 12.8 ms
  2. 写入新位置:   256 pages × 1500μs = 384 ms (TLC)
  3. 擦除旧块:     1 block × 5ms = 5 ms
  总计: ~402 ms

如果每分钟执行 1 次:
  额外 I/O 时间: 402ms / 60s = 0.67%
  对平均性能影响: < 1%（可接受）
  
  但如果与用户 I/O 并发:
  可能导致 P99 延迟增加 ~400ms（不可接受！）
  → 应在空闲时执行静态 WL
```

## 4. 车规级 15 年寿命 WL 策略

### 4.1 寿命计算

```
产品参数:
  容量: 128 GB
  NAND 类型: TLC, P/E = 3000 次
  OP 比例: 12.5%
  Raw 容量: 144 GB

每日写入量 (DWPD) 估算:
  ADAS 录像: 50 MB/s × 8h = 1.44 TB/天
  系统日志: ~10 GB/天
  导航缓存: ~5 GB/天
  OTA 升级: ~2 GB/天（均摊）
  总计: ~1.46 TB/天

写放大因子 (WAF): 假设 WAF = 2.0

实际 NAND 写入量: 1.46 TB × 2.0 = 2.92 TB/天

每日擦除次数:
  2.92 TB / 144 GB = ~20.3 次/天（全盘均匀分布时）

15 年总擦除次数:
  20.3 × 365 × 15 = ~111,000 次

TLC P/E 限制: 3000 次

结论: 111,000 / 3000 = 37 倍超限！
      ADAS 场景下 128GB TLC 无法满足 15 年寿命
      
解决方案:
  1. 增大容量（256GB+），降低每块擦除频率
  2. 使用更高 P/E 的 NAND（车规级 TLC 5000+）
  3. 优化写入策略（减少 WAF）
  4. 限制 ADAS 写入量（降码率/循环覆盖）
  5. 磨损均衡确保均匀分布（必须！）
```

### 4.2 车规 WL 策略要点

1. **严格的擦除均匀度**: EC 差值 < 5%（全生命周期）
2. **静态 WL 必须启用**: 冷数据不能永久占据低 EC 块
3. **WL 元数据掉电保护**: 擦除计数表必须持久化
4. **温度感知 WL**: 高温加速 NAND 退化，高温区域的块应降低使用优先级
5. **健康监控**: 通过 Health Descriptor 实时报告寿命状态

## 5. Read Disturb 和 Data Retention

### 5.1 Read Disturb

频繁读取一个页会干扰同一块中其他页的数据：

```
Block 内部:
  Page 0: [数据正常]  ← 被频繁读取
  Page 1: [数据正常]  ← 受到读干扰
  Page 2: [数据正常]  ← 受到读干扰
  ...
  Page 255: [数据正常] ← 受到读干扰

读取次数超过阈值（如 100K-1M 次）后:
  Page 1-255 的数据可能出现 bit flip
  需要通过 ECC 纠正或主动刷新（Read Refresh）
```

**WL 关联**：高读取频率的块需要更频繁地刷新（读取→重写到新块→擦除旧块），这增加了该块的擦除次数，WL 需要将这些额外的擦除计入均衡策略。

### 5.2 Data Retention

NAND 存储的数据会随时间自然退化：

| NAND 类型 | P/E=0 时保持时间 | P/E=Max 时保持时间 | 温度影响 |
|-----------|-----------------|-------------------|----------|
| SLC | 10+ 年 | 1+ 年 | 高温加速退化 |
| TLC | 3-5 年 | 3-12 个月 | 高温加速退化 |
| QLC | 1-3 年 | 1-6 个月 | 高温严重加速 |

**车规影响**：车辆可能长时间停放（数月），高温环境（夏天车内 60-80°C）加速数据退化。WL 策略需要考虑：
- 高擦除次数的块保持时间更短
- 需要定期刷新（Data Refresh）长期未读写的块

## 6. 测试方法

### 6.1 WL 均匀度验证

```bash
#!/bin/bash
# 磨损均衡验证测试
DEVICE="/dev/sdX"
UFS_BSG="/dev/ufs-bsg0"

# 1. 记录初始擦除计数
echo "=== Initial Erase Count ==="
ufs-utils attr -r -p $UFS_BSG -a bDeviceLifeTimeEstA
ufs-utils attr -r -p $UFS_BSG -a bDeviceLifeTimeEstB

# 2. 执行偏斜写入（只写 10% 的地址空间）
fio --name=skewed_write \
    --filename=$DEVICE \
    --rw=randwrite --bs=4k \
    --size=12G \
    --numjobs=1 --iodepth=32 \
    --runtime=3600 --time_based

# 3. 检查擦除分布
# 通过厂商工具读取各块擦除计数
# 计算: EC_max, EC_min, EC_avg, EC_stddev
# 验收: (EC_max - EC_min) / EC_avg < 10%
```

### 6.2 静态 WL 触发验证

```bash
# 1. 写入冷数据（不再更新）
dd if=/dev/urandom of=$DEVICE bs=1M count=64000 oflag=direct

# 2. 持续热写入 10% 区域
fio --name=hot_write \
    --filename=$DEVICE \
    --rw=randwrite --bs=4k \
    --offset=0 --size=12G \
    --numjobs=1 --iodepth=32 \
    --runtime=7200 --time_based

# 3. 如果静态 WL 工作正常:
#    冷数据块的擦除次数应该逐渐增加
#    （冷数据被迁移到高 EC 块）
```

### 6.3 寿命加速测试

| 测试项 | 方法 | 时长 | 验证指标 |
|--------|------|------|----------|
| 擦除均匀度 | 偏斜写入 1000 小时 | ~42 天 | EC 差值 < 10% |
| 寿命估算 | 全盘连续写入 | 直到 LifeTimeEst 变化 | 寿命与预期一致 |
| 高温加速 | 85°C + 持续写入 | 500 小时 | 功能正常 |
