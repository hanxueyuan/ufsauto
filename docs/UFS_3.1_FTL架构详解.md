## 1. FTL 概述

### 1.1 什么是 FTL

FTL（Flash Translation Layer，闪存转换层）是 UFS 固件中最核心的软件模块。它在 Host 看到的逻辑地址空间（LBA）和 NAND Flash 的物理地址空间（PBA）之间建立映射关系，同时管理 NAND 的特殊约束：

- **先擦后写**：NAND 不能原地覆盖，必须先擦除整个块再写入
- **擦除粒度大**：擦除以块（Block）为单位（通常 2-16 MB），写入以页（Page）为单位（通常 4-16 KB）
- **有限 P/E 循环**：每个块的擦写次数有限（TLC: 1000-3000 次，QLC: 500-1000 次）
- **读干扰**：频繁读取会干扰相邻页的数据

### 1.2 FTL 在 UFS 系统中的位置

```
┌──────────────────────────────────────────────────┐
│  UFS Host                                        │
│  (SCSI READ/WRITE → LBA 地址)                    │
└────────────────────┬─────────────────────────────┘
                     │ UPIU (UFS Protocol)
┌────────────────────┴─────────────────────────────┐
│  UFS Device Controller (群联 PS8363)              │
│  ┌───────────────────────────────────────────┐   │
│  │            FTL (Flash Translation Layer)   │   │
│  │  ┌─────────┬──────────┬────────┬────────┐ │   │
│  │  │ 地址映射 │ 垃圾回收  │磨损均衡│坏块管理│ │   │
│  │  │ Mapping │    GC    │  WL   │  BBM   │ │   │
│  │  └─────────┴──────────┴────────┴────────┘ │   │
│  │           ↕ 映射表 (L2P Table)              │   │
│  │           ↕ 元数据管理                      │   │
│  └───────────────────────────────────────────┘   │
│  ┌───────────────────────────────────────────┐   │

│  │         (长存 SQS 3D TLC NAND)             │   │
│  └───────────────────────────────────────────┘   │
└──────────────────────────────────────────────────┘
```

## 2. 三种映射方式

### 2.1 页映射（Page-level Mapping）

**原理**：每个逻辑页（LPN）独立映射到一个物理页（PPN）。

```
逻辑地址空间          映射表 (L2P)          物理地址空间
┌─────────┐         ┌───────────┐        ┌──────────┐
│ LPN 0   │ ──────→ │ PPN 1024  │ ─────→ │ Block 4  │
│ LPN 1   │ ──────→ │ PPN 2048  │ ─────→ │ Page 0   │
│ LPN 2   │ ──────→ │ PPN 512   │ ─────→ │ Block 2  │
│ ...      │         │ ...       │        │ ...      │
│ LPN N-1 │ ──────→ │ PPN 8192  │ ─────→ │ Block 32 │
└─────────┘         └───────────┘        └──────────┘
```

**优点**：
- 随机写入性能最优（任意 LPN 可映射到任意 PPN）
- 不需要合并操作（Merge-free）
- GC 效率高

**缺点**：
- 映射表巨大，占用大量 RAM

**128GB UFS 映射表大小计算**：
```
总容量:        128 GB = 137,438,953,472 bytes
页大小:        4 KB = 4,096 bytes
总逻辑页数:    128 GB / 4 KB = 33,554,432 (32M 页)
映射条目大小:  4 bytes (32-bit PPN)
映射表大小:    32M × 4B = 128 MB

结论: 页映射需要 128MB RAM 存储映射表
     对于嵌入式控制器来说成本偏高
```

### 2.2 块映射（Block-level Mapping）

**原理**：只维护块级别的映射，块内偏移保持不变。

```
逻辑地址空间          映射表 (L2B)          物理地址空间
┌─────────┐         ┌─────────��─┐        ┌──────────┐
│ LBN 0   │ ──────→ │ PBN 4     │ ─────→ │ Block 4  │
│  Page 0 │         │           │        │  Page 0  │
│  Page 1 │         │           │        │  Page 1  │
│  ...     │         │           │        │  ...     │
│  Page 255│         │           │        │  Page 255│
├─────────┤         ├───────────┤        ├──────────┤
│ LBN 1   │ ──────→ │ PBN 7     │ ─────→ │ Block 7  │
└─────────┘         └───────────┘        └──────────┘
```

**映射表大小计算**（128GB）：
```
块大小:        2 MB (256 pages × 4KB + OOB)
总逻辑块数:    128 GB / 2 MB = 65,536 (64K 块)
映射条目大小:  4 bytes
映射表大小:    64K × 4B = 256 KB

结论: 块映射只需 256KB RAM，但随机写性能极差
```

**缺点**：
- 随机写入需要 Read-Modify-Write（读整块→修改→写入新块→擦除旧块）
- 严重的写放大

### 2.3 混合映射（Hybrid Mapping）— 主流方案

**原理**：结合页映射和块映射的优势。大部分数据用块映射，少量热数据用页映射（Log Block）。

```
┌─────────────────────────────────────────────┐
│              混合映射架构                      │
├─────────────────────────────────────────────┤
│                                             │
│  数据块区域（Data Block）：块映射            │
│  ┌─────┬─────┬─────┬─────┬─────┐          │
│  │DB 0 │DB 1 │DB 2 │ ... │DB N │          │
│  └─────┴─────┴─────┴─────┴─────┘          │
│                                             │
│  日志块区域（Log Block）：页映射             │
│  ┌─────┬─────┬─────┬─────┐                │
│  │LB 0 │LB 1 │LB 2 │LB 3 │ (数量有限)    │
│  └─────┴─────┴─────┴─────┘                │
│                                             │
│  写入流程：                                  │
│  1. 随机写入 → 先写入 Log Block（页映射）    │
│  2. Log Block 满 → 合并到 Data Block        │
│  3. 顺序写入 → 直接写入 Data Block          │
└─────────────────────────────────────────────┘
```

**FAST（Fully Associative Sector Translation）方案**：
- Log Block 可服务任意 Data Block
- 灵活性高，但合并开销大

**DFTL（Demand-based FTL）方案**（现代主流）：
- 全页映射，但只缓存热点映射条目到 RAM
- 冷映射条目存储在 NAND 的专用区域（CMT: Cached Mapping Table）
- 按需加载：Cache Miss 时从 NAND 读取映射条目
- 典型缓存命中率 > 95%

```
RAM 中的映射缓存（CMT）:
┌───────────────────────────┐
│ LPN → PPN (热点条目)       │  大小: 8-32 MB
│ 命中率: >95%               │
└────────────┬──────────────┘
             │ Cache Miss
             ↓
NAND 中的完整映射表（GMT）:
┌───────────────────────────┐
│ 全量 L2P 映射表            │  大小: 128 MB
│ 存储在 NAND 专用区域       │
└───────────────────────────┘
```

## 3. 地址转换流程

### 3.1 读取流程（LBA → 数据）

```
Host 发送 READ_10 (LBA=0x1000, Length=1)
         │
         ↓
┌─────────────────────────┐
│ 1. LBA → LPN 转换       │  LPN = LBA / (PageSize/SectorSize)
│    LPN = 0x1000 / 8 = 512│  (假设 4KB 页, 512B 扇区)
└────────┬────────────────┘
         ↓
┌─────────────────────────┐
│ 2. 查找映射表            │  在 CMT 中查找 LPN=512
│    CMT Hit?              │
└──┬──────────────┬───────┘
   │ Hit          │ Miss
   ↓              ↓
┌──────────┐  ┌──────────────────┐
│ 读取 PPN │  │ 从 NAND 加载映射 │
│ PPN=2048 │  │ 更新 CMT          │
└────┬─────┘  └────────┬─────────┘
     │                  │
     ↓                  ↓
┌─────────────────────────┐
│ 3. PPN → 物理地址解码    │
│    Block = PPN / 256     │  Block = 8
│    Page  = PPN % 256     │  Page = 0
│    Die   = Block / ...   │
│    Plane = ...           │
└────────┬────────────────┘
         ↓
┌─────────────────────────┐
│ 4. NAND Read             │  读取 Die 0, Block 8, Page 0
│    + ECC 纠错            │
│    + 返回数据给 Host     │
└─────────────────────────┘
```

### 3.2 写入流程（数据 → NAND）

```
Host 发送 WRITE_10 (LBA=0x1000, Data)
         │
         ↓
┌─────────────────────────┐
│ 1. 数据写入 Write Buffer │  先缓存到 SRAM/DRAM
│    (Write Buffer: 4-32MB)│
└────────┬────────────────┘
         ↓
┌─────────────────────────┐
│ 2. 分配新物理页          │  从 Free Page Pool 获取空闲页
│    新 PPN = 3072         │
└────────┬────────────────┘
         ↓
┌─────────────────────────┐
│ 3. 写入 NAND             │
│    NAND Program:         │
│    Die 0, Block 12,      │
│    Page 0                │
└────────┬────────────────┘
         ↓
┌─────────────────────────┐
│ 4. 更新映射表            │
│    旧: LPN 512 → PPN 2048│  标记旧 PPN 2048 为无效
│    新: LPN 512 → PPN 3072│  更新 CMT
└────────┬────────────────┘
         ↓
┌─────────────────────────┐
│ 5. 返回成功              │
│    SCSI Status = GOOD    │
└─────────────────────────┘
```

## 4. SLC Cache / Write Buffer 机制

### 4.1 SLC Cache 原理

现代 UFS 设备使用 TLC/QLC NAND，但通过 SLC 模式编程来加速写入：

```
TLC NAND（每个 Cell 存 3 bits）:
  编程时间: ~1500 μs
  写入带宽: ~300 MB/s

SLC 模式（同一 Cell 只存 1 bit）:
  编程时间: ~200 μs
  写入带宽: ~1000 MB/s
  代价: 容量减少为 1/3
```

### 4.2 UFS 3.1 Write Booster

JESD220E 定义了 Write Booster (WB) 功能：

| 参数 | 典型值 | 说明 |
|------|--------|------|
| WB Buffer 大小 | 2-10 GB | SLC 模式的缓存区域 |
| WB 类型 | Shared / Dedicated | 共享 LUN / 独立 LUN |
| WB 启用 | bWriteBoosterEn 属性 | Host 可控 |
| WB Flush | 自动 / Host 触发 | SLC → TLC 数据搬移 |

**SLC Cache 生命周期**：

```
阶段1: SLC Cache 空（性能最高）
  写入速度: ~1000 MB/s
  ↓ 持续写入
  
阶段2: SLC Cache 使用中
  写入速度: ~1000 MB/s（仍在 SLC 区域）
  后台: SLC → TLC Flush 正在进行
  ↓ 写入速度 > Flush 速度
  
阶段3: SLC Cache 满（Write Cliff）
  写入速度: 骤降到 ~200-300 MB/s
  原因: 必须等待 Flush 释放空间或直接写 TLC
  ↓ 停止写入，等待 Flush
  
阶段4: SLC Cache 恢复
  后台 Flush 完成，SLC 空间释放
  写入速度: 恢复到 ~1000 MB/s
```

## 5. 掉电安全与映射表备份

### 5.1 掉电风险

UFS 设备在以下时刻掉电最危险：

| 掉电时刻 | 风险 | 后果 |
|----------|------|------|
| NAND 编程中 | 页数据不完整 | 该页数据丢失 |
| 映射表更新中 | 映射表不一致 | 地址映射错误 |
| GC 执行中 | 有效页迁移不完整 | 数据丢失 |
| SLC→TLC Flush 中 | 数据搬移不完整 | SLC 数据仍在，可恢复 |

### 5.2 掉电保护机制

**1. 映射表日志（Journal）**：
```
正常更新流程:
  1. 将映射变更写入 Journal（顺序追加）
  2. 定期将 Journal 合并到主映射表
  3. 掉电恢复时：回放 Journal 恢复最新映射

Journal 结构:
  ┌──────────────────────────┐
  │ Entry 1: LPN=512→PPN=3072│
  │ Entry 2: LPN=100→PPN=4096│
  │ Entry 3: LPN=888→PPN=5120│
  │ ...                      │
  └──────────────────────────┘
```

**2. 映射表双备份**：
```
NAND 中的映射表存储:
  ┌──────────────┐  ┌──────────────┐
  │ 映射表 Copy A │  │ 映射表 Copy B │
  │ (主)          │  │ (备)          │
  └──────────────┘  └──────────────┘
  
  更新策略: A/B 交替写入
  掉电恢复: 读取最新的完整备份
```

**3. 原子写入保证**：
```
UFS 控制器确保:
  - 每个 4KB 页的写入是原子的
  - 要么完整写入，要么完全不写
  - 不会出现半写页（Partial Page Program）
  
实现方式:
  - 片内电容（Capacitor）提供短暂供电
  - 足以完成当前页的编程
  - 典型保持时间: 1-5 ms
```

### 5.3 车规级掉电安全要求

| 要求 | 标准 | 说明 |
|------|------|------|
| 数据持久化 | 已确认写入的数据不丢失 | SYNCHRONIZE_CACHE 后的数据 |
| 映射表恢复 | 掉电后映射表可恢复 | < 1 秒恢复时间 |
| 元数据一致性 | 掉电不导致元数据损坏 | Journal + 双备份 |
| 坏块表保护 | 掉电不导致 BBT 丢失 | BBT 多副本 |
| 启动时间 | 掉电恢复后快速启动 | < 3 秒（车规要求） |

## 6. Host-aware FTL vs Device-managed FTL

### 6.1 Device-managed FTL（当前主流）

```
Host 视角:   只看到 LBA 地址空间，完全不知道 FTL 内部
Device 视角: 完全自主管理 FTL，Host 无法干预

优点: Host 驱动简单，兼容性好
缺点: Host 无法优化 I/O 模式，GC 不可预测
```

### 6.2 Host-aware FTL（UFS 3.1 新增）

UFS 3.1 引入了一些 Host-aware 特性：

```
1. Write Booster 控制:
   Host 可以启用/禁用 Write Booster
   Host 可以触发 WB Flush

2. HPB (Host Performance Booster):
   Host 缓存部分 L2P 映射表
   减少设备端映射查找延迟
   
3. UNMAP/Trim:
   Host 通知设备哪些 LBA 不再使用
   设备可以提前回收这些空间
   减少 GC 压力
```

## 7. 128GB UFS 3.1 FTL 参数估算

### 长存 SQS 3D TLC NAND 参数

| 参数 | 值 | 说明 |
|------|-----|------|
| Die 数量 | 4 | 4 个 32GB Die |
| Plane/Die | 2 | 双平面 |
| Block/Plane | 2048 | |
| Page/Block | 256 | TLC: 3 × lower/middle/upper |
| Page 大小 | 16 KB | |
| Block 大小 | 4 MB | 256 × 16KB |
| OOB/Page | 1.2 KB | ECC + 元数据 |
| P/E Cycle | 3000 | TLC 典型值 |
| 读延迟 | 50-75 μs | |
| 编程延迟 | 700-1500 μs | TLC 模式 |
| 擦除延迟 | 3-5 ms | |

### FTL 资源估算

```
总物理块数:     4 Die × 2 Plane × 2048 Block = 16,384 块
总物理容量:     16,384 × 4 MB = 64 GB（raw）
                实际 128GB 产品 raw 容量 ~144 GB
                
Over-Provisioning: (144 - 128) / 128 ≈ 12.5%
                   OP 空间用于: GC 空间 + 坏块替换 + 映射表存储

映射表大小:     
  页映射 (16KB 页): 128 GB / 16 KB × 4B = 32 MB
  DFTL CMT 缓存:   8-16 MB (覆盖 25-50% 热点)
  
Free Block Pool:  ~5% 总块数 = ~820 块
Reserved Blocks:  ~2% 用于坏块替换 = ~330 块
```
