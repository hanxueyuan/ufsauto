# UFS 3.1 车规项目 - 代码审查指南

## 一、审查范围
所有提交到 UFS 3.1 车规项目的代码变更，包括但不限于：
- 功能实现代码
- 测试用例
- 文档更新
- 配置文件修改
- 构建脚本变更

## 二、审查标准

### 1. 功能正确性
- [ ] 代码实现符合需求规格说明书要求
- [ ] 边界条件处理正确
- [ ] 异常处理完整且合理
- [ ] 没有逻辑错误
- [ ] 与现有功能无冲突

### 2. 车规安全合规性（ISO 26262 ASIL B 要求）
- [ ] 符合 MISRA C:2012 编码规范（C 代码）
- [ ] 符合 Python 车规安全编码规范
- [ ] 没有未定义行为
- [ ] 内存操作安全（无缓冲区溢出、野指针等）
- [ ] 关键路径有冗余校验
- [ ] 故障安全机制完善

### 3. 代码质量
- [ ] 代码简洁、易读
- [ ] 命名规范统一（遵循项目命名约定）
- [ ] 注释清晰且必要
- [ ] 函数/模块职责单一
- [ ] 避免重复代码
- [ ] 符合项目架构设计要求

### 4. 性能要求
- [ ] 读写性能符合 UFS 3.1 规范要求（顺序读≥2100MB/s，顺序写≥1600MB/s）
- [ ] 响应延迟符合车规要求（≤10ms）
- [ ] 内存占用在设计范围内
- [ ] CPU 使用率合理
- [ ] 无性能瓶颈

### 5. 测试覆盖率
- [ ] 单元测试覆盖率 ≥ 80%
- [ ] 关键功能路径覆盖率 100%
- [ ] 测试用例覆盖正常场景和异常场景
- [ ] 测试用例可重复执行
- [ ] 所有测试通过

### 6. 兼容性
- [ ] 兼容 ARM v7/v8 架构
- [ ] 兼容主流 UFS 3.1 控制器
- [ ] 向后兼容 UFS 2.1 设备
- [ ] 符合车规级硬件兼容性要求

## 三、审查流程

### 1. 审查时限
- 普通 PR：24 小时内完成审查
- 紧急 PR：4 小时内完成审查
- 重大变更 PR：48 小时内完成审查

### 2. 审查步骤
1. 自动检查：CI 流水线自动运行静态代码检查、单元测试、覆盖率检查
2. 人工审查：代码审查工程师进行全面审查
3. 问题反馈：对发现的问题提出修改意见
4. 代码修改：开发者根据反馈修改代码
5. 二次审查：对修改后的代码进行重新审查
6. 审查通过：标记 PR 可合并

### 3. 审查结果分类
- ✅ 通过：代码符合所有审查标准，可以合并
- ⚠️ 建议：非强制性改进建议，开发者可选择性采纳
- ❌ 不通过：存在严重问题，需要修改后重新提交审查

## 四、常见问题 Checklist

### C 代码专项检查
- [ ] 没有使用 goto 语句（除了错误处理场景）
- [ ] 所有变量初始化
- [ ] 数组边界检查
- [ ] 指针使用前判空
- [ ] 内存分配/释放配对
- [ ] 没有使用魔数
- [ ] 函数返回值检查
- [ ] 循环有明确的终止条件
- [ ] 没有递归调用
- [ ] 中断处理函数简洁

### Python 代码专项检查
- [ ] 符合 PEP 8 规范
- [ ] 类型注解完整
- [ ] 异常捕获范围合理
- [ ] 资源使用 with 语句管理
- [ ] 没有全局变量滥用
- [ ] 日志记录规范
- [ ] 输入参数校验
- [ ] 避免使用 eval/exec 等危险函数

### ARM 架构专项检查
- [ ] 内存对齐正确
- [ ] 端序处理正确
- [ ] 硬件寄存器操作安全
- [ ] 缓存一致性处理
- [ ] 中断屏蔽/恢复正确
- [ ] DMA 操作符合规范

## 五、编码规范
### 1. 命名规范
- 文件名：小写字母+下划线，如 `ufs_core.c`
- 函数名：小驼峰式，如 `ufsReadSector`
- 变量名：小驼峰式，如 `sectorCount`
- 宏定义：大写字母+下划线，如 `UFS_SECTOR_SIZE`
- 常量名：大写字母+下划线，如 `MAX_TRANSFER_SIZE`

### 2. 注释规范
- 函数头注释：说明功能、参数、返回值、注意事项
- 复杂代码块注释：说明实现逻辑
- 关键变量注释：说明变量含义和取值范围
- 避免不必要的注释

### 3. 格式规范
- 缩进：4 个空格
- 行宽：不超过 120 个字符
- 大括号：独占一行
- 运算符前后留空格
