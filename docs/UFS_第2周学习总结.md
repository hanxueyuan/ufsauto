## 2. 关键知识收获

### 2.1 ICC 控制（核心新知识）

**关键认知**：
1. ICC 不是简单的限流，而是 Host 告知 Device 电源系统能力的协议机制
2. 16 个级别覆盖从"不消耗电流"(Level 0) 到"设备最大能力"(Level 15)
3. ICC Level 直接影响 NAND Die 并行度，进而影响写入性能
4. 车载场景需要根据系统状态动态调整 ICC Level（电池电量/温度/负载）
5. ICC 切换应避免在 I/O 密集期间执行

**测试关键点**：
- ICC Level vs 性能曲线（找到性能拐点）
- ICC Level vs 功耗曲线（确认线性关系）
- ICC 切换对正在执行命令的影响
- POWERDOWN 唤醒后 ICC Level 是否保持

### 2.2 QoS 指标体系

**关键认知**：
1. 车规级应用中 QoS 比峰值性能更重要——ADAS 不能容忍延迟尖峰
2. P99/P99.99 延迟主要由 GC（垃圾回收）引起
3. Write Cliff 是 UFS 最重要的性能现象——SLC Cache 耗尽后性能骤降 3-5 倍
4. 稳态性能（Steady State）才是真实性能，需要按 SNIA PTS 方法测量
5. 不同车载应用（ADAS/仪表/IVI/EDR）QoS 需求差异巨大

**验收标准关键数据**：
- 顺序读 ≥ 1500 MB/s（128KB, QD32）
- 稳态顺序写 ≥ 200 MB/s（全盘预写后）
- 读延迟 P99 ≤ 500μs（4KB, QD1）
- 写延迟 P99.99 ≤ 50ms（稳态）
- 最大延迟 < 100ms（绝对红线）

### 2.3 电源模式转换时序

**关键认知**：
1. HIBERN8 唤醒时间的瓶颈是 M-PHY PLL 重新锁定（占 60-80%）
2. SLEEP 和 HIBERN8 在物理层都是 M-PHY HIBERN8，区别在逻辑层状态保持
3. 三层协同转换：应用层→UniPro→M-PHY，每层都有超时机制
4. Linux ufshcd 通过 Runtime PM 管理 HIBERN8，默认 3 秒空闲后进入
5. 链路错误恢复有明确的升级路径：重传→链路重置→Host重置
