## 1. 为什么需要垃圾回收

### 1.1 NAND 的先擦后写特性

NAND Flash 不能像 RAM 一样原地覆盖数据。写入流程：

```
写入新数据到已有 LBA:
  ❌ 不能: 直接覆盖旧物理页
  ✅ 必须: 写入新物理页 → 标记旧页为无效 → 等待 GC 回收

结果: 随着时间推移，每个块中混合了有效页和无效页
```

```
Block 状态示例（256 pages/block）:
┌────────────────────────────────────┐
│ Block 5                            │
│ Page 0:   [有效] ← 最新数据        │
│ Page 1:   [无效] ← 已被覆盖写入    │
│ Page 2:   [有效]                   │
│ Page 3:   [无效]                   │
│ Page 4:   [无效]                   │
│ Page 5:   [有效]                   │
│ ...                                │
│ Page 255: [有效]                   │
│                                    │
│ 有效页: 128/256 = 50%             │
│ 无效页: 128/256 = 50%             │
│ 这些无效页占用的空间无法直接使用！  │
└────────────────────────────────────┘

GC 的任务: 
  1. 将 128 个有效页拷贝到其他块
  2. 擦除整个 Block 5
  3. Block 5 变成空闲块，可重新使用
```

### 1.2 空间碎片化

随着使用时间增长，Free Block 逐渐减少：

```
初始状态:     Free Blocks = 1000, Used Blocks = 0
使用一段时间: Free Blocks = 50,  Used Blocks = 950
                                 (每块约 50% 无效页)

实际可用空间 = 50 blocks × 4MB + 950 blocks × 50% × 4MB
             = 200 MB + 1900 MB = 2100 MB

但 Free Block 只剩 50 个 = 200 MB 连续可写空间
→ 必须 GC 回收无效空间！
```

## 2. 前台 GC（Foreground GC）

### 2.1 触发条件

当 Free Block 数量低于**紧急阈值**时触发前台 GC：

```
Free Block 阈值:
  正常:     Free > 100 blocks → 不触发 GC
  后台 GC:  Free < 100 blocks → 启动后台 GC
  前台 GC:  Free < 20 blocks  → 强制前台 GC（阻塞写入！）
  紧急:     Free < 5 blocks   → 紧急模式（可能拒绝写入）
```

### 2.2 执行流程

```
写入请求到达
     │
     ↓
┌─────────────────────────┐
│ 检查 Free Block 数量     │
│ Free = 15 < 20 (阈值)   │
└────────┬────────────────┘
         ↓
┌─────────────────────────┐
│ 暂停用户写入             │  ← 这就是延迟尖峰的来源！
│ (Host 的写入命令等待)     │
└────────┬────────────────┘
         ↓
┌─────────────────────────┐
│ 选择 Victim Block        │
│ (选择无效页最多的块)      │
│ Block 5: 有效率 30%      │ ← 选中
│ Block 8: 有效率 50%      │
│ Block 12: 有效率 70%     │
└────────┬────────────────┘
         ↓
┌─────────────────────────┐
│ 拷贝有效页               │
│ Block 5 的 77 个有效页   │
│ → 写入到空闲块           │
│ 耗时: 77 × 1500μs       │
│      = 115.5 ms          │
└────────┬────────────────┘
         ↓
┌──────��──────────────────┐
│ 擦除 Block 5             │
│ 耗时: ~5 ms              │
└────────┬────────────────┘
         ↓
┌─────────────────────────┐
│ Block 5 加入 Free Pool   │
│ Free = 16                │
│ 恢复用户写入             │
└─────────────────────────┘

总延迟: ~120 ms（这就是 P99 延迟尖峰！）
```

### 2.3 对性能的影响

```
正常写入延迟:  ~200 μs (SLC Cache) / ~1500 μs (TLC 直写)
前台 GC 延迟:  50-500 ms（取决于 Victim 块的有效页数量）

影响:
  P50 延迟: 不受影响（大部分写入不触发 GC）
  P99 延迟: 可能增加 10-100 倍
  P99.99:   可能达到 500ms+

车规影响:
  ADAS 录像: 单帧延迟 > 50ms 可能导致丢帧
  数据记录: 延迟 > 100ms 可能丢失关键事件数据
```

## 3. 后台 GC（Background GC）

### 3.1 触发条件

设备空闲时自动执行，不影响用户 I/O：

```
后台 GC 触发条件:
  1. Free Block < 后台阈值（如 100 blocks）
  2. 设备处于 IDLE 状态（无用户命令）
  3. 或用户 I/O 间隙（命令间隔 > 1ms）
```

### 3.2 执行策略

```
后台 GC 策略:
  1. 低优先级执行（用户 I/O 立即打断 GC）
  2. 每次只回收 1 个块（减少单次耗时）
  3. 选择有效率最低的块（回收效率最高）
  4. 在 IDLE 和 SLEEP 模式间歇执行

时序:
  用户 I/O ──────────────── 空闲 ── 用户 I/O ─── 空闲
                              │                     │
                        后台 GC (1块)          后台 GC (1块)
```

### 3.3 后台 GC 的局限

- 如果写入速率持续很高（如 ADAS 持续录像），设备可能没有空闲时间执行后台 GC
- 最终 Free Block 耗尽，不得不触发前台 GC
- **车规对策**：设计合理的写入 Duty Cycle，保证设备有足够的空闲时间执行后台 GC

## 4. Victim Block 选择算法

### 4.1 Greedy 算法（贪心）

选择**无效页最多**（有效率最低）的块：

```
算法: 选择 Valid_Pages 最少的块
优点: 回收效率最高（拷贝的有效页最少）
缺点: 不考虑块的擦除次数（可能加剧磨损不均）

选择过程:
  Block 5:  Valid = 30/256 = 11.7%  ← 选中
  Block 8:  Valid = 128/256 = 50%
  Block 12: Valid = 200/256 = 78.1%
```

### 4.2 Cost-Benefit 算法

综合考虑有效率和块的年龄（自上次写入以来的时间）：

```
公式: Benefit/Cost = (1 - u) × age / (2u)

其中:
  u = 有效页比例 (valid_pages / total_pages)
  age = 自上次修改以来的时间

选择 Benefit/Cost 最大的块

优点: 平衡了回收效率和数据稳定性
      老数据更可能是冷数据，回收后不会很快再产生无效页
缺点: 需要维护每个块的时间戳，增加元数据开销
```

### 4.3 CAT���Cost-Age-Times）算法

在 Cost-Benefit 基础上加入擦除次数因素：

```
公式: Score = (1 - u) × age / (2u × EC_normalized)

EC_normalized = 该块擦除次数 / 平均擦除次数

效果: 高擦除次数的块得分更低，减少被选为 Victim 的概率
      → 自带磨损均衡效果
```

### 4.4 算法对比

| 算法 | 回收效率 | WL 效果 | 复杂度 | 适用场景 |
|------|----------|---------|--------|----------|
| Greedy | 最高 | 无 | O(n) | 简单系统 |
| Cost-Benefit | 高 | 中等 | O(n) | 通用 |
| CAT | 中高 | 好 | O(n) | 车规级（推荐） |

## 5. 写放大因子（WAF）

### 5.1 定义

```
WAF = 实际 NAND 写入量 / Host 写入量

WAF = 1.0: 完美效率（Host 写 1GB，NAND 写 1GB）
WAF = 2.0: NAND 写入是 Host 的 2 倍（GC 额外写入 1 倍）
WAF = 3.0: 严重写放大
```

### 5.2 WAF 的组成

```
WAF_total = WAF_gc + WAF_wl + WAF_metadata

WAF_gc:       GC 有效页拷贝产生的额外写入
WAF_wl:       WL 数据迁移产生的额外写入
WAF_metadata: 映射表/日志等元数据写入

典型值:
  随机写入（4KB, 无 OP）: WAF ≈ 5-10
  随机写入（4KB, 12% OP）: WAF ≈ 2-4
  顺序写入:                 WAF ≈ 1.0-1.2
  混合负载（70R/30W）:      WAF ≈ 2-3
```

### 5.3 WAF 与 OP 的关系

Over-Provisioning（预留空间）直接影响 WAF：

```
OP = 7%:   WAF ≈ 3.5 (随机写)
OP = 12%:  WAF ≈ 2.5
OP = 28%:  WAF ≈ 1.8
OP = 50%:  WAF ≈ 1.3

原因: OP 越大 → Free Block 越多 → GC 触发频率越低 
      → 每次 GC 的 Victim 块有效率越低 → 拷贝的有效页越少
```

### 5.4 WAF 对寿命的影响

```
设备寿命 = (容量 × P/E 循环) / (每日写入量 × WAF)

示例 (128GB TLC, P/E=3000):
  WAF=2.0: 寿命 = 128GB × 3000 / (100GB/天 × 2.0) = 1920 天 ≈ 5.3 年
  WAF=3.0: 寿命 = 128GB × 3000 / (100GB/天 × 3.0) = 1280 天 ≈ 3.5 年
  WAF=1.5: 寿命 = 128GB × 3000 / (100GB/天 × 1.5) = 2560 天 ≈ 7.0 年

结论: WAF 每降低 0.5，寿命增加约 30%
```

## 6. GC 对 P99 延迟的影响机制

### 6.1 延迟尖峰产生过程

```
时间轴:

正常写入 ─────── GC触发 ────────────── 正常写入
200μs 200μs 200μs │                    │ 200μs 200μs
                   │ 前台GC执行          │
                   │ 有效页拷贝: 100ms   │
                   │ 块擦除: 5ms         │
                   │ 映射更新: 1ms       │
                   │ 总计: ~106ms        │
                   └────────────────────┘

在这 106ms 期间:
  - Host 的写入命令排队等待
  - 等待的命令延迟 = 106ms + 原始延迟
  - 这些命令就是 P99/P99.99 的延迟尖峰
```

### 6.2 车规级确定性延迟策略

```
策略1: 预留更多 OP
  → 减少前台 GC 触发频率
  → 代价: 可用容量减少

策略2: 限制前台 GC 每次处理量
  → 每次只回收 1 个块（而非多个）
  → 将一次大延迟分散为多次小延迟
  → 最大单次延迟: ~50ms（可控）

策略3: I/O 调度器感知 GC
  → Host 在检测到延迟增加时降低写入速率
  → 给后台 GC 更多执行时间

策略4: 分区隔离
  → ADAS 数据写入独立分区
  → 该分区使用更高 OP 比例
  → GC 不影响其他分区
```

## 7. 测试方法

### 7.1 触发 GC 的测试方法

```bash
#!/bin/bash
# 强制触发 GC 测试
DEVICE="/dev/sdX"

# 步骤1: 全盘写入（耗尽 Free Blocks）
echo "=== 步骤1: 全盘写入 ==="
fio --name=fill --filename=$DEVICE --rw=write --bs=128k --size=100% --iodepth=32

# 步骤2: 随机覆盖写入（产生大量无效页）
echo "=== 步骤2: 随机覆盖写入 ==="
fio --name=overwrite --filename=$DEVICE \
    --rw=randwrite --bs=4k --size=100% \
    --numjobs=1 --iodepth=32 \
    --runtime=300 --time_based \
    --lat_percentiles=1 \
    --percentile_list=50:90:99:99.9:99.99 \
    --output=gc_test_result.json --output-format=json

# 步骤3: 分析延迟分布
echo "=== 延迟分布 ==="
python3 -c "
import json
with open('gc_test_result.json') as f:
    data = json.load(f)
job = data['jobs'][0]
clat = job['write']['clat_ns']['percentile']
print(f'P50:    {clat.get(\"50.000000\", 0)/1e6:.2f} ms')
print(f'P99:    {clat.get(\"99.000000\", 0)/1e6:.2f} ms')
print(f'P99.9:  {clat.get(\"99.900000\", 0)/1e6:.2f} ms')
print(f'P99.99: {clat.get(\"99.990000\", 0)/1e6:.2f} ms')
print(f'Max:    {job[\"write\"][\"lat_ns\"][\"max\"]/1e6:.2f} ms')
"
```

### 7.2 GC 性能影响量化

| 测试场景 | 方法 | 观察指标 |
|----------|------|----------|
| GC 前 | 新盘顺序写入 | 基准带宽/延迟 |
| GC 触发 | 全盘写满后随机写 | P99 延迟增幅 |
| GC 恢复 | 停止写入等待 N 秒后测试 | 性能恢复时间 |
| 持续 GC | 稳态随机写 1 小时 | 稳态 P99 延迟 |
