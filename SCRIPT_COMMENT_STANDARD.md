# 测试脚本注释规范

## 🎯 注释要求
所有测试脚本开头必须包含标准化的头部注释，清晰说明测试的目的、步骤、预期结果等信息，方便人类阅读和维护。

---
## 📋 标准注释模板
```python
#!/usr/bin/env python3
"""
测试目的：[一句话说明这个测试要验证什么功能/性能/可靠性指标]
测试模块：[sys/func/perf/rel/scen，对应模块缩写]
测试用例ID：[符合命名规则的用例ID，例如：t_func_large_file_rw_002]
测试优先级：[P0/P1/P2，P0最高]
前置条件：[运行这个测试需要满足的前提条件，例如：可用空间≥10GB、UFS设备已识别]
测试步骤：
    1. [第一步操作，例如：在测试目录下创建10个1GB大小的测试文件]
    2. [第二步操作，例如：计算每个文件的md5校验和]
    3. [第三步操作，例如：读取文件，重新计算md5并对比]
预期结果：
    1. [第一步的预期结果，例如：文件创建成功，无IO错误]
    2. [第二步的预期结果，例如：md5计算正常完成]
    3. [第三步的预期结果，例如：读取成功，md5校验一致]
测试耗时：[预计运行时间，例如：约10分钟]
特殊说明：[其他需要说明的内容，例如：会占用大量磁盘空间、需要root权限等]

修改记录：
    2026-03-12 张三 初始版本
    2026-03-15 李四 优化测试逻辑，增加错误处理
    2026-03-20 王五 调整测试参数，适配新的硬件环境
"""
```

## ✅ 示例
```python
#!/usr/bin/env python3
"""
测试目的：验证大文件读写功能和数据完整性
测试模块：func
测试用例ID：t_func_large_file_rw_002
测试优先级：P0
前置条件：
    1. /mapdata/ufs_test目录可用空间≥20GB
    2. 运行用户有读写权限
测试步骤：
    1. 在测试目录下创建10个2GB大小的随机数据文件
    2. 记录每个文件的md5校验和
    3. 依次读取每个文件，重新计算md5校验和
    4. 对比两次md5是否一致
预期结果：
    1. 所有文件创建成功，无IO错误
    2. 所有文件读取成功，无IO错误
    3. 两次md5校验和完全一致，数据无损坏
测试耗时：约15分钟
特殊说明：测试完成后会自动删除临时文件，不会占用持久空间

修改记录：
    2026-03-12 测试开发工程师B 初始版本，支持10个2GB文件测试
    2026-03-16 测试开发工程师B 优化文件创建速度，提升测试效率30%
    2026-03-22 QA工程师 增加错误处理，记录失败时的系统日志
"""
```

## 📌 注释要求
### 1. 必须包含的字段
- ✅ 测试目的：清晰说明测试的目标和验证的内容
- ✅ 测试模块：对应模块缩写，方便分类
- ✅ 测试用例ID：符合命名规则的唯一ID
- ✅ 测试优先级：明确测试的重要程度
- ✅ 前置条件：运行测试需要满足的前提
- ✅ 测试步骤：详细的操作步骤，任何人看了都能手动复现
- ✅ 预期结果：每一步对应的预期结果
- ✅ 修改记录：每次修改都要记录修改人、时间、修改内容

### 2. 可选字段
- 测试耗时：预计的运行时间，方便测试计划安排
- 特殊说明：需要注意的事项，例如资源占用、权限要求等

### 3. 更新要求
- 每次修改脚本功能、步骤、参数时，必须同步更新头部注释
- 修改记录要写清楚修改的内容和原因，方便追溯
- 注释要和代码保持一致，禁止出现注释和代码不匹配的情况

---
## ❌ 错误示例
### 1. 注释过于简单
```python
"""
测试大文件读写
"""
```
❌ 缺少模块、ID、步骤、预期结果等必要信息，其他人无法理解测试的具体内容。

### 2. 注释和代码不匹配
```python
"""
测试步骤：
    1. 创建10个1GB文件
    2. 计算md5
    3. 对比校验和
"""
# 实际代码创建的是20个2GB文件
```
❌ 注释和代码不一致，会误导使用者。

### 3. 没有修改记录
修改了测试逻辑但没有更新注释和修改记录，无法追溯变更历史。

---
## 🔍 检查机制
1. 代码审查时QA工程师会重点检查注释是否符合规范
2. CI/CD流水线会自动扫描头部注释，缺少必要字段的代码无法提交
3. 定期抽查注释和代码的一致性，发现不匹配的情况要求立即修正
