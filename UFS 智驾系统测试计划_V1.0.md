**文档版本**: V1.1  
**创建时间**: 2026-03-13  
**修订时间**: 2026-03-13  
**测试对象**: 车载 UFS 存储系统  
**适用场景**: 智能驾驶系统存储验证

---

## 📌 一、测试目标

本测试计划旨在全面验证 UFS 在智驾系统环境下的：
- **性能表现**：稳态性能、延迟特性、缓存效率
- **功能完整性**：UFS 标准功能、Multi-LUN、电源管理
- **可靠性**：上下电、异常掉电、长时间运行
- **环境适应性**：高低温、振动、车载工况

---

## 📌 二、测试核心思想

### 2.1 性能测试设计理念

> **关键洞察**：性能测试不只是测"快不快"，更要测"为什么快/慢"

- **稳态性能**：避免瞬时性能假象，验证长时间负载下的真实表现
- **空闲梯度分析**：通过空闲时间扫描（20ms~30s），反推固件后台任务触发机制
- **缓存分析**：分离 SLC Cache 和 Data Cache 的效果，通过性能对比间接推断缓存效果
- **QoS 延迟**：智驾对延迟敏感，关注尾延迟（P99/P99.99）而非平均延迟

### 2.2 可靠性测试设计理念

> **关键洞察**：车载环境的特殊性决定了可靠性测试必须覆盖极端工况

- **上下电循环**：模拟车辆频繁启动/熄火场景
- **异常掉电**：模拟车祸、电瓶故障等意外断电
- **电源状态切换**：验证低功耗模式下的功能完整性
- **长时间运行**：30 天连续运行，验证无性能衰减、无掉盘

### 2.3 固件算法分析理念

> **关键洞察**：通过性能表现反推固件算法策略

| 触发阈值 | 恢复时间 | 性能下降幅度 | 固件策略推断 |
|----------|----------|--------------|--------------|
| 20~40ms | <1s | >30% | **非常激进**：快速启动、快速完成，优先长期性能 |
| 40~60ms | 1~5s | 20~30% | **激进**：较快启动后台整理 |
| 60~100ms | 5~10s | 10~20% | **适中**：平衡响应速度和整理效率 |
| 100ms~1s | 10~30s | <10% | **保守**：延迟启动，优先响应速度 |
| >1s | >30s | 微小 | **非常保守**：Lazy GC 策略 |

#### 3.1.3 空闲/活跃性能分析（固件算法反推）

**测试目的**：通过空闲时间梯度扫描，反推固件后台任务触发机制

**测试流程**：
```
步骤 1：建立活跃状态
  └── 连续写入 1 小时，使 UFS 进入稳定活跃状态

步骤 2：空闲梯度扫描（关键区间加密）
  
  【关键区间】20ms~100ms（固件后台任务触发敏感区，10ms 步长精确定位）
  ├── 空闲 20ms → 读取测试 → 记录性能
  ├── 空闲 30ms → 读取测试 → 记录性能
  ├── 空闲 40ms → 读取测试 → 记录性能
  ├── 空闲 50ms → 读取测试 → 记录性能
  ├── 空闲 60ms → 读取测试 → 记录性能
  ├── 空闲 80ms → 读取测试 → 记录性能
  └── 空闲 100ms → 读取测试 → 记录性能
  
  【中等区间】100ms~1s（验证任务执行过程）
  ├── 空闲 200ms → 读取测试 → 记录性能
  ├── 空闲 500ms → 读取测试 → 记录性能
  └── 空闲 1s → 读取测试 → 记录性能
  
  【秒级区间】1s~30s（验证任务完成时间）
  ├── 空闲 2s → 读取测试 → 记录性能
  ├── 空闲 5s → 读取测试 → 记录性能
  ├── 空闲 10s → 读取测试 → 记录性能
  └── 空闲 30s → 读取测试 → 记录性能
  
  【恢复验证】验证性能完全恢复
  ├── 空闲 1 分钟 → 读取测试 → 记录性能
  └── 空闲 10 分钟 → 读取测试 → 记录性能

步骤 3：分析性能拐点
  └── 找出性能开始下降的时间点 → 后台任务触发阈值
  └── 找出性能最低点 → 后台任务执行峰值
  └── 找出性能恢复到 95% 的时间点 → 后台任务完成时间
```

**关键指标**：
- 后台任务触发阈值（XX ms）：性能开始下降的空闲时间点
- 性能下降幅度（XX%）：性能谷底的下降比例
- 性能恢复时间（XX s）：性能恢复到 95% 以上的空闲时间
- 固件策略推断（激进/适中/保守）

**固件策略判断表**：

| 触发阈值 | 恢复时间 | 性能下降幅度 | 固件策略推断 |
|----------|----------|--------------|--------------|
| 20~40ms | <1s | >30% | **非常激进**：快速启动、快速完成，优先长期性能 |
| 40~60ms | 1~5s | 20~30% | **激进**：较快启动后台整理 |
| 60~100ms | 5~10s | 10~20% | **适中**：平衡响应速度和整理效率 |
| 100ms~1s | 10~30s | <10% | **保守**：延迟启动，优先响应速度 |
| >1s | >30s | 微小 | **非常保守**：Lazy GC 策略 |

**测试点设计理由**：

| 时间区间 | 测试点 | 设计理由 |
|----------|--------|----------|
| **关键区间** | 20/30/40/50/60/80/100ms | 固件后台任务触发敏感区，10ms 步长精确定位触发阈值 |
| **中等区间** | 200/500ms/1s | 验证任务执行过程中的性能变化曲线 |
| **秒级区间** | 2/5/10/30s | 验证任务完成时间和性能恢复情况 |
| **恢复验证** | 1min/10min | 确认性能完全恢复，排除其他因素影响 |

**总计测试点**：16 个（关键区间 7 个 + 中等区间 3 个 + 秒级区间 4 个 + 恢复验证 2 个）

#### 3.1.4 Data Cache 性能测试

**测试目的**：验证 Data Cache 的存在、容量和效果

**注意**：缓存命中率无法直接测量（固件黑盒，无对外接口），通过冷热数据性能对比间接推断缓存效果。

| 测试项 | 测试方法 | 可测量指标 | 验收标准 |
|--------|----------|------------|----------|
| 缓存大小测试 | 写入不同大小文件 (128MB/256MB/512MB/1GB/2GB/4GB)，立即读取对比速度 | 读取速度/写入速度比值 | 缓存内读取速度≥写入速度 3 倍；找到"速度骤降"临界点推断缓存容量 |
| 冷热数据对比 | 首次随机读取 10000 个 4K 块（冷）vs 重复读取同样数据（热） | 热数据延迟/冷数据延迟 | 热数据读取延迟≤冷数据 50%，间接证明缓存生效 |
| 缓存刷新机制 | 写入 1GB 数据后，不同时间点 (1s/10s/1min/10min/1h) 读取 | 性能衰减时间曲线 | 明确刷新时间窗口（如 10s~1min 内性能骤降说明缓存刷新） |
| 缓存开关对比 | 如固件支持配置，对比开启/关闭缓存的性能 | 性能提升比例 | 缓存开启时性能提升≥30% |

**测试流程示例（缓存大小测试）**：
```
1. 依次写入文件：128MB → 256MB → 512MB → 1GB → 2GB → 4GB
2. 每个文件写入完成后立即读取（1 秒内）
3. 记录每次的写入速度和读取速度
4. 计算读取/写入速度比
5. 找到"读取速度骤降"的临界点 → 推断缓存容量边界

预期现象：
- 文件≤缓存容量：读取速度 >> 写入速度（如 3 倍以上）
- 文件>缓存容量：读取速度 ≈ 写入速度（缓存已绕过）
```

**关键指标**：
- 缓存容量边界（GB）
- 缓存内读取速度提升比例（%）
- 缓存刷新时间窗口（s/min）
- 热数据延迟降低比例（%）

## 📌 六、安全保障措施

1. **测试文件控制**：所有性能测试采用固定大小测试文件（最大 20GB）
2. **自动清理**：测试完成后自动删除测试文件
3. **空间保护**：测试不会写满分区，保留至少 20% 可用空间
4. **数据备份**：关键测试前备份重要数据
5. **异常监控**：测试过程中实时监控温度、错误计数，异常时自动停止
